# Product Surface Plan: Product Catalog Management System

## Planning Summary

- **Problem / Opportunity**: Administrators need an efficient, secure, and scalable system to manage product inventory, categories, and catalog visibility without manual processes or fragmented tools.
- **Primary Users / Personas**: Internal administrators (editors, managers, admins) managing product data for a public-facing storefront.
- **Business Outcome**: Reduce catalog management time by 50%, eliminate manual errors in product data, enable real-time catalog updates, and support growth to 1M+ products.
- **Recommended Delivery Shape**: MVP then increments — deliver core CRUD and authentication first, then add advanced features (bulk operations, custom attributes, exports) in subsequent releases.
- **Formal Design Readiness**: Ready with Assumptions

## Scope Boundaries

### In Scope
- Web-based admin panel for product and category management
- Secure authentication and role-based access control (RBAC)
- Full CRUD operations on products and categories
- Image upload, storage, and CDN delivery
- Product search, filtering, and inventory tracking
- Audit logging of all administrative actions
- Soft-delete and hard-delete with confirmation
- Product visibility control (draft/published)
- Bulk operations on products (edit, delete, categorize)
- Category hierarchy with drag-and-drop reordering
- Custom product attributes per category
- Catalog export (CSV/JSON)
- Responsive web UI for desktop and tablet
- AWS-only infrastructure

### Out of Scope
- Public-facing storefront (consumes catalog via API but is a separate application)
- Mobile native apps (iOS/Android) for admin users
- Customer-facing features (reviews, ratings, wishlists)
- Payment processing or order management
- Multi-language or multi-currency support (unless specified later)
- Real-time collaboration features (e.g., live cursors, chat)
- Advanced analytics or BI dashboards (basic audit logs only)
- Third-party marketplace integrations (e.g., Amazon, eBay)

### Assumptions
- Public storefront exists separately and consumes product data via REST or GraphQL API
- Admins access the system via modern browsers on desktop or tablet devices
- Initial catalog size is under 100,000 products; system must scale to 1M+
- Category hierarchy will not exceed 5 levels deep
- Admin team size will not exceed 50 concurrent users
- Product images are provided in JPG, PNG, or WebP formats, max 50MB per file, max 20 images per product
- Data residency requirements are known (e.g., us-east-1 or eu-west-1) but not specified in requirements — **Assumption Needed**
- Budget allows for managed AWS services without aggressive cost optimization constraints
- Admins are trained on basic inventory management concepts
- No existing product catalog system is being replaced — **Greenfield application**

## Product Flow Overview

1. **Admin Authentication**: Admin navigates to admin panel URL, enters email/password, completes MFA challenge (if enabled), receives session token from Cognito, redirected to dashboard.
2. **Dashboard Load**: Web UI fetches product list from backend API (paginated), displays products in table/grid view, shows category tree in sidebar, loads user permissions from Cognito token.
3. **Product Creation**: Admin clicks "Add Product", fills form (name, description, price, SKU, images), uploads images to S3 via presigned URL, submits form to backend API, backend validates SKU uniqueness, saves product to RDS, triggers image processing Lambda, returns success response, UI refreshes product list.
4. **Product Editing**: Admin clicks "Edit" on product, loads product details from API, modifies fields, uploads new images or reorders existing, submits changes, backend checks for concurrent edits (optimistic locking), saves changes to RDS, logs edit to DynamoDB audit log, invalidates CloudFront cache, returns success response, UI updates product list.
5. **Category Management**: Admin navigates to "Categories" tab, views hierarchical category tree, drags category to reorder or change parent, submits reorder request to API, backend updates category positions in RDS, returns success response, UI updates tree view.
6. **Product Search/Filter**: Admin enters search query or selects filters (category, price range), web UI sends search request to API, backend queries RDS with full-text search or ElastiCache, returns filtered product list, UI displays results with pagination.
7. **Bulk Operations**: Admin selects multiple products via checkboxes, chooses bulk action (edit, delete, categorize), confirms action in modal, web UI sends bulk request to API, backend enqueues jobs to SQS, Lambda workers process jobs asynchronously, UI polls for progress updates, displays completion notification.
8. **Audit Log Review**: Admin navigates to "Audit Logs" tab, filters logs by date/user/action, web UI fetches logs from DynamoDB via API, displays paginated log entries, admin exports logs to CSV if needed.

## Domain Concepts and Data Needs

### Domain Data Needs
| Domain Concept / Object | Storage Need | Key Attributes | Relationships / Ownership | Retention / Lifecycle |
| --- | --- | --- | --- | --- |
| Product | SQL (RDS PostgreSQL) | id, name, description, price, sku, status (draft/published), stock, created_at, updated_at, deleted_at, version | Belongs to multiple categories (many-to-many); owned by creator (user_id) | Soft-deleted products retained for 90 days; hard-deleted products removed immediately; active products retained indefinitely |
| Category | SQL (RDS PostgreSQL) | id, name, parent_id, order, created_at, updated_at | Self-referential (parent-child hierarchy); has many products (many-to-many) | Retained indefinitely; deleted categories cascade to product-category assignments |
| ProductCategory | SQL (RDS PostgreSQL) | product_id, category_id | Join table for many-to-many relationship | Deleted when product or category is deleted |
| ProductImage | SQL (RDS PostgreSQL) + Object Storage (S3) | id, product_id, image_url, order, created_at | Belongs to product; image file stored in S3 | Images retained as long as product exists; deleted when product is hard-deleted |
| AuditLog | NoSQL (DynamoDB) | id, user_id, action, resource_type, resource_id, changes (JSON), timestamp | Belongs to user; references product, category, or other resource | Retained for 2 years in DynamoDB; archived to S3 after 1 year for cost optimization |
| User | SQL (RDS PostgreSQL) or Cognito | id, email, name, role (editor/manager/admin), mfa_enabled, created_at, updated_at | Owns products, categories, audit logs | Retained indefinitely; deactivated users marked as inactive |
| BulkOperationJob | NoSQL (DynamoDB) | job_id, user_id, action, product_ids, params, status, progress, created_at, updated_at | Belongs to user; references products | Retained for 30 days; completed jobs archived to S3 |

### Storage Decisions
- **Primary Persistence Need**: Relational database (RDS PostgreSQL) for products, categories, product-category relationships, product images metadata, and users. PostgreSQL chosen for strong relational model support, ACID transactions, full-text search, and JSON column support for custom attributes.
- **Object/File Storage**: S3 for product images (original and resized versions); CloudFront for CDN delivery; presigned URLs for secure uploads.
- **Cache / Search / Analytics**: ElastiCache (Redis) for caching product list queries, category tree, and search results (optional, if PostgreSQL full-text search is insufficient); OpenSearch considered for advanced search but deferred to later phase.
- **Migrations**: Use database migration tool (e.g., Flyway, Liquibase, or Alembic) for schema versioning; initial schema includes products, categories, product_categories, product_images, users tables; audit logs in DynamoDB (no schema migration needed).

## Cross-Surface Contracts

- **Shared API Schema**: OpenAPI 3.0 specification for all REST endpoints; auto-generated from backend code (e.g., FastAPI, NestJS); versioned (`/api/v1/`); published to S3 + CloudFront for web UI and storefront consumption.
- **Error Response Format**: Consistent JSON error format: `{ "error": { "code": "VALIDATION_ERROR", "message": "SKU already exists", "details": { "field": "sku", "value": "ABC123" } } }`; HTTP status codes: 400 (validation), 401 (unauthorized), 403 (forbidden), 404 (not found), 409 (conflict), 500 (server error).
- **Pagination and Filtering Conventions**: Query params: `page` (1-indexed), `limit` (default 50, max 1000), `search` (full-text search), `category_id`, `min_price`, `max_price`, `status` (draft/published); response includes `total`, `page`, `limit`, `has_next`, `has_prev`.
- **Authentication/Session Behavior**: Cognito JWT tokens in `Authorization: Bearer <token>` header; tokens expire after 30 minutes; refresh tokens valid for 7 days; web UI refreshes tokens automatically; API returns 401 if token expired.
- **File Upload/Download Flow**: Web UI requests presigned URL from `POST /api/v1/images/presigned-url`; uploads image directly to S3 using presigned URL; S3 triggers Lambda for processing; Lambda updates product_images table; web UI polls `GET /api/v1/products/:id` for image metadata.
- **Versioning Strategy for Mobile Clients**: Not applicable (no mobile app in scope).
- **Backward-Compatible Contract Changes**: Not applicable (greenfield; no existing clients to preserve).
- **Deprecation Timelines and Telemetry**: Not applicable (greenfield).
