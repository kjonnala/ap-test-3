# Solution Design

## Summary

This design document provides a comprehensive, implementation-ready architecture for the **Product Catalog Management System** — a secure, scalable admin platform for managing product inventory, categories, and catalog visibility on AWS.

### What's Included

This design synthesizes multiple architectural concerns into one cohesive solution:

1. **Single Page Application (SPA) Frontend Design** — React-based admin panel with TypeScript, Tailwind CSS, and Zustand for state management
2. **REST API Backend Design** — Node.js/Express API with Lambda or ECS Fargate, enforcing RBAC and business rules
3. **Relational Database Design** — PostgreSQL schema for products, categories, images, and audit trails
4. **AWS Infrastructure Architecture** — Cognito, API Gateway, Lambda, RDS, S3, CloudFront, DynamoDB, SQS, EventBridge
5. **Cross-Cutting Concerns** — Authentication, authorization, observability, error handling, performance optimization

## Technology Stack & Architecture Pattern

### Frontend Stack
- **Framework**: React 18 with TypeScript
- **Build Tool**: Vite (fast dev server, optimized production builds)
- **Styling**: Tailwind CSS (utility-first, responsive design)
- **State Management**: Zustand (lightweight global state) + TanStack Query (server state/cache)
- **Routing**: React Router v6 (declarative routing, lazy loading)
- **Forms**: React Hook Form + Zod (validation schema)
- **API Client**: Axios with interceptors for auth and error handling
- **Testing**: Vitest + React Testing Library (unit and integration tests)
- **Accessibility**: Radix UI primitives (WCAG 2.1 AA compliant components)

### Backend Stack
- **Runtime**: Node.js 20 LTS
- **Framework**: Express.js (REST API)
- **Language**: TypeScript
- **Compute**: AWS Lambda (serverless, auto-scaling) or ECS Fargate (containerized, long-running)
- **API Gateway**: AWS API Gateway (REST API, throttling, caching, custom domain)
- **Authentication**: AWS Cognito (user pool, MFA, JWT tokens)
- **Validation**: Zod (schema validation for request/response)
- **ORM**: Prisma (type-safe database client, migrations)
- **Testing**: Jest + Supertest (unit and integration tests)

### Database & Storage
- **Primary Database**: Amazon RDS PostgreSQL 15 (Multi-AZ, automated backups)
- **Audit Logs**: Amazon DynamoDB (append-only, point-in-time recovery)
- **Object Storage**: Amazon S3 (product images, exports)
- **CDN**: Amazon CloudFront (global image delivery, caching)
- **Cache**: Amazon ElastiCache (Redis) — optional for search/filter caching

### Infrastructure & DevOps
- **IaC**: AWS CDK (TypeScript) or Terraform
- **CI/CD**: GitHub Actions or AWS CodePipeline + CodeBuild + CodeDeploy
- **Monitoring**: Amazon CloudWatch (logs, metrics, dashboards, alarms)
- **Tracing**: AWS X-Ray (distributed tracing)
- **Security**: AWS WAF (DDoS protection), AWS Secrets Manager (credentials), AWS KMS (encryption keys)
- **Messaging**: Amazon SQS (FIFO queues for bulk operations)
- **Scheduler**: Amazon EventBridge (scheduled jobs for alerts, exports, backups)

### Architecture Pattern

**Modular Monolith with Serverless Components**

- **Web UI**: Single-page React app hosted on S3 + CloudFront (static assets)
- **API Layer**: Express.js REST API running on Lambda (serverless) or ECS Fargate (containerized)
- **Service Layer**: Domain services (ProductService, CategoryService, AuditLogService) encapsulate business logic
- **Data Layer**: Prisma ORM for PostgreSQL access; DynamoDB SDK for audit logs
- **Background Jobs**: Lambda functions triggered by SQS, S3 events, or EventBridge schedules
- **API Gateway**: Single entry point for all API requests; handles auth, throttling, caching

**Why Modular Monolith?**
- Simpler deployment and debugging for MVP
- Shared domain logic and database transactions
- Easy to extract microservices later if needed (e.g., ImageProcessingService, BulkOperationService)
- Reduced operational complexity compared to microservices

### Page Designs

#### DashboardPage (`/`)
- **Route**: `/`
- **Purpose**: Overview of product catalog with quick stats and product list
- **Components**: StatsCards, ProductList (table), QuickActions
- **State**: Product list (paginated, filtered), search query, selected filters
- **API Calls**: `GET /api/v1/products`, `GET /api/v1/categories`
- **User Interactions**: Search products, apply filters, navigate to create/edit, select products for bulk actions

#### LoginPage (`/login`)
- **Route**: `/login`
- **Purpose**: Admin authentication with email/password and MFA
- **Components**: LoginForm, MFAChallenge
- **State**: Email, password, MFA code, loading state, error message
- **API Calls**: `POST /auth/login` (Cognito)
- **User Interactions**: Enter credentials, submit, complete MFA challenge, redirect to dashboard

#### ProductCreatePage (`/products/new`)
- **Route**: `/products/new`
- **Purpose**: Create new product with all required fields and images
- **Components**: ProductForm (BasicInfoSection, CategorySelector, ImageUpload, InventorySection, VisibilityToggle)
- **State**: Form data (name, description, SKU, price, categories, images, stock, status), validation errors, loading state
- **API Calls**: `POST /api/v1/images/presigned-url`, `POST /api/v1/products`
- **User Interactions**: Fill form, upload images, select categories, submit, handle validation errors, redirect to product detail on success

#### ProductEditPage (`/products/:id/edit`)
- **Route**: `/products/:id/edit`
- **Purpose**: Edit existing product with optimistic locking
- **Components**: ProductForm (pre-filled with existing data)
- **State**: Form data, validation errors, loading state, version (for optimistic locking)
- **API Calls**: `GET /api/v1/products/:id`, `PUT /api/v1/products/:id`, `POST /api/v1/images/presigned-url`
- **User Interactions**: Modify fields, upload/reorder images, submit, handle concurrent edit conflicts (409 Conflict), redirect to product detail on success

#### ProductDetailPage (`/products/:id`)
- **Route**: `/products/:id`
- **Purpose**: View product details, images, and audit history
- **Components**: ProductInfo, ImageGallery, AuditHistory, ActionButtons
- **State**: Product data, audit logs (paginated)
- **API Calls**: `GET /api/v1/products/:id`, `GET /api/v1/audit-logs?product_id=:id`
- **User Interactions**: View details, navigate to edit, delete product (with confirmation), view audit history

#### CategoriesPage (`/categories`)
- **Route**: `/categories`
- **Purpose**: Manage category hierarchy with drag-and-drop reordering
- **Components**: CategoryTree, CategoryForm (modal), CategoryActions

- **API Calls**: `GET /api/v1/categories`, `POST /api/v1/categories`, `PUT /api/v1/categories/:id`, `DELETE /api/v1/categories/:id`

#### AuditLogsPage (`/audit-logs`)
- **Route**: `/audit-logs`

- **API Calls**: `GET /api/v1/audit-logs`, `POST /api/v1/audit-logs/export`

### State Management Design

#### Global State (Zustand)

| State Slice | Data | Update Triggers | Persistence |
|---|---|---|---|
| `authStore` | `user`, `token`, `refreshToken`, `roles`, `isAuthenticated` | Login, logout, token refresh | localStorage (encrypted) |
| `uiStore` | `sidebarOpen`, `theme`, `notifications` | User interactions, system events | localStorage |

#### Server State (TanStack Query)

**Queries** (cached data fetched from API):
- `useProducts` — `GET /api/v1/products` (paginated, filtered)
- `useProduct` — `GET /api/v1/products/:id`
- `useCategories` — `GET /api/v1/categories`
- `useAuditLogs` — `GET /api/v1/audit-logs` (paginated, filtered)
- `useBulkStatus` — `GET /api/v1/products/bulk/:job_id/status`

**Mutations** (data modifications):
- `useCreateProduct` — `POST /api/v1/products`
- `useUpdateProduct` — `PUT /api/v1/products/:id`
- `useDeleteProduct` — `DELETE /api/v1/products/:id`
- `useCreateCategory` — `POST /api/v1/categories`
- `useUpdateCategory` — `PUT /api/v1/categories/:id`
- `useDeleteCategory` — `DELETE /api/v1/categories/:id`
- `useBulkOperation` — `POST /api/v1/products/bulk`

**Cache Strategy**:
- **Stale-while-revalidate**: Product list, category tree (stale after 5 minutes, refetch in background)
- **Cache invalidation**: On mutation success, invalidate related queries (e.g., create product → invalidate product list)
- **Optimistic updates**: Update UI immediately on mutation, rollback on error

#### Form State (React Hook Form + Zod)

- **ProductForm**: `name`, `description`, `price`, `sku`, `category_ids`, `images`, `stock`, `low_stock_threshold`, `status`, `version`
- **CategoryForm**: `name`, `parent_id`, `order`
- **AuditLogFilters**: `start_date`, `end_date`, `user_id`, `action`, `product_id`

**Validation**:
- Client-side: Zod schema validation (required fields, SKU format, price range, image file size/type)
- Server-side: API enforces business rules (SKU uniqueness, category hierarchy depth, role permissions)

### API Integration Layer

#### API Service Structure

**Base Configuration** (`src/services/api.ts`):
```typescript
import axios from 'axios';
import { authStore } from '@/stores/authStore';

const api = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL, // e.g., https://api.example.com
  timeout: 30000,
  headers: {
    'Content-Type': 'application/json',
  },
});

// Request interceptor: Inject auth token
api.interceptors.request.use((config) => {
  const token = authStore.getState().token;
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// Response interceptor: Handle errors and token refresh
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      // Token expired, attempt refresh
      const refreshToken = authStore.getState().refreshToken;
      if (refreshToken) {
        try {
          const { data } = await axios.post('/auth/refresh', { refreshToken });
          authStore.getState().setToken(data.access_token);
          // Retry original request
          error.config.headers.Authorization = `Bearer ${data.access_token}`;
          return api.request(error.config);
        } catch (refreshError) {
          // Refresh failed, logout
          authStore.getState().logout();
          window.location.href = '/login';
        }
      }
    }
    return Promise.reject(error);
  }
);

export default api;
```

**Error Handling Strategy**:
- **Network errors**: Show toast notification, retry button
- **Validation errors (400)**: Display inline error messages on form fields
- **Unauthorized (401)**: Attempt token refresh, logout if refresh fails
- **Forbidden (403)**: Show permission denied message
- **Not found (404)**: Show "resource not found" message
- **Conflict (409)**: Show concurrent edit conflict message, prompt to refresh
- **Server errors (500)**: Show generic error message, log to monitoring

**Retry Logic**:
- Retry on network errors (max 3 retries with exponential backoff)
- No retry on 4xx errors (client errors)
- Retry on 5xx errors (server errors, max 3 retries)

**Protected Route Implementation**:
```typescript
const ProtectedRoute = ({ children, requiredRole }: { children: React.ReactNode, requiredRole?: string }) => {
  const { isAuthenticated, roles } = authStore();
  
  if (!isAuthenticated) {
    return <Navigate to="/login" />;
  }
  
  if (requiredRole && !roles.includes(requiredRole)) {
    return <PermissionDenied />;
  }
  
  return <>{children}</>;
};
```

### Design Decisions & Trade-offs

**Frontend Decisions**:

1. **React over Vue/Angular**: React has the largest ecosystem, best TypeScript support, and most AWS Amplify integrations.
2. **Vite over Create React App**: Vite offers faster dev server, optimized builds, and better DX.
3. **Zustand over Redux Toolkit**: Zustand is simpler, less boilerplate, and sufficient for this app's state complexity.
4. **TanStack Query over manual fetch**: Automatic caching, refetching, and optimistic updates reduce boilerplate and improve UX.
5. **Tailwind CSS over CSS-in-JS**: Tailwind is faster, smaller bundle size, and easier to maintain for utility-first design.
6. **React Hook Form over Formik**: Better performance (uncontrolled inputs), smaller bundle, TypeScript-first.
7. **Radix UI over Material-UI**: Unstyled primitives allow custom design, better accessibility, smaller bundle.

**Trade-offs**:
- **Client-side validation only**: Faster UX, but server-side validation is still enforced for security.
- **Optimistic updates**: Better perceived performance, but requires rollback logic on error.
- **Lazy loading**: Reduces initial bundle size, but adds slight delay when navigating to new pages.

## Backend API Design

### Service Scope

**Responsibilities**:
- Authenticate and authorize admin users via AWS Cognito
- Enforce role-based access control (RBAC) for all operations
- Validate and persist product, category, and audit log data
- Generate presigned URLs for secure image uploads to S3
- Process bulk operations asynchronously via SQS + Lambda
- Trigger background jobs for image processing, malware scanning, alerts, and exports
- Provide REST API for web UI and public storefront consumption

**Non-Goals**:
- Public-facing storefront UI (separate application)
- Customer-facing features (reviews, ratings, wishlists)
- Payment processing or order management
- Real-time collaboration features (live cursors, chat)
- Advanced analytics or BI dashboards

**External Dependencies**:
- AWS Cognito (authentication, MFA, user management)
- AWS RDS PostgreSQL (product, category, user data)
- AWS S3 (product images, exports)
- AWS CloudFront (CDN for images)
- AWS DynamoDB (audit logs, bulk operation job status)
- AWS SQS (bulk operation job queue)
- AWS Lambda (image processing, malware scanning, alerts, exports)
- AWS EventBridge (scheduled jobs)

### Cross-Cutting Design

#### Authentication and Authorization

**Authentication** (AWS Cognito):
- Cognito user pool stores admin users (email, password, MFA settings)
- Login flow: `POST /auth/login` → Cognito `InitiateAuth` → returns JWT access token and refresh token
- MFA flow: If MFA enabled, Cognito returns `MFA_REQUIRED` challenge → user submits MFA code → Cognito returns tokens
- Token refresh: `POST /auth/refresh` → Cognito `InitiateAuth` with refresh token → returns new access token
- Token expiry: Access tokens expire after 30 minutes; refresh tokens valid for 7 days

**Authorization** (RBAC):
- Cognito groups: `editors`, `managers`, `admins`
- API Gateway authorizer validates Cognito JWT tokens
- Lambda functions check user role (from Cognito groups claim) and enforce permissions server-side
- Permission matrix:

| Action | Editor | Manager | Admin |
|---|---|---|---|
| View products | ✅ | ✅ | ✅ |
| Create product | ✅ | ✅ | ✅ |
| Edit product | ✅ | ✅ | ✅ |
| Soft-delete product | ❌ | ✅ | ✅ |
| Hard-delete product | ❌ | ❌ | ✅ |
| Bulk operations | ❌ | ✅ | ✅ |
| View audit logs | ❌ | ❌ | ✅ |
| Manage users | ❌ | ❌ | ✅ |

**Implementation**:
```typescript
// Middleware to check user role
const requireRole = (allowedRoles: string[]) => {
  return (req: Request, res: Response, next: NextFunction) => {
    const userRoles = req.user.roles; // From Cognito JWT claims
    const hasPermission = allowedRoles.some(role => userRoles.includes(role));
    if (!hasPermission) {
      return res.status(403).json({ error: { code: 'FORBIDDEN', message: 'Permission denied' } });
    }
    next();
  };
};

// Usage
router.delete('/api/v1/products/:id', requireRole(['admin']), productController.hardDelete);
```

### Deployment Considerations

**Statelessness**:
- API is stateless; all session state stored in Cognito JWT tokens
- No in-memory session storage; allows horizontal scaling

**Configuration and Secrets**:
- Environment variables for non-sensitive config (API Gateway URL, S3 bucket name, RDS endpoint)
- AWS Secrets Manager for sensitive config (RDS password, Cognito client secret, API keys)
- Secrets rotated automatically via Secrets Manager rotation Lambda

**Horizontal Scaling and Performance Hotspots**:
- **Lambda**: Auto-scales to handle concurrent requests; monitor cold start latency
- **ECS Fargate** (if used): Auto-scaling based on CPU/memory utilization
- **RDS**: Use read replicas for read-heavy queries (product list, search)
- **ElastiCache**: Cache product list, category tree, search results to reduce RDS load

**Performance Hotspots**:

## Database Design

### Data Model Overview

**Core Domains**:
- **Product Management**: Products, categories, product-category relationships, product images
- **User Management**: Admin users, roles, permissions (stored in Cognito and RDS)
- **Audit Trail**: Immutable logs of all administrative actions (stored in DynamoDB)
- **Background Jobs**: Bulk operation job status, progress tracking (stored in DynamoDB)

**Primary Ownership Boundaries**:
- Products owned by creator (user_id)
- Categories owned by creator (user_id)
- Audit logs owned by user who performed action (user_id)
- Bulk operation jobs owned by user who initiated job (user_id)

### Entities and Relationships

| Entity | Purpose | Primary Key | Relationships |
|---|---|---|---|
| `products` | Store product data (name, description, price, SKU, stock, status) | `id` (UUID) | Belongs to multiple categories (many-to-many via `product_categories`); has many images (one-to-many via `product_images`); owned by user (foreign key `created_by`) |
| `categories` | Store category hierarchy (name, parent, order) | `id` (UUID) | Self-referential (parent-child via `parent_id`); has many products (many-to-many via `product_categories`) |
| `product_categories` | Join table for product-category many-to-many relationship | `(product_id, category_id)` | Belongs to product and category |
| `product_images` | Store product image metadata (URL, order) | `id` (UUID) | Belongs to product (foreign key `product_id`) |
| `users` | Store admin user metadata (name, email, role) | `id` (UUID) | Owns products, categories, audit logs; synced from Cognito |
| `audit_logs` (DynamoDB) | Immutable log of all administrative actions | `id` (UUID) | References user, product, category, or other resource |
| `bulk_operation_jobs` (DynamoDB) | Track bulk operation job status and progress | `job_id` (UUID) | Belongs to user; references products |

### Schema Notes

#### `products` Table

```sql
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  description TEXT,
  price DECIMAL(10, 2) NOT NULL CHECK (price >= 0),
  sku VARCHAR(100) NOT NULL UNIQUE,
  status VARCHAR(20) NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published')),
  stock INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0),
  low_stock_threshold INTEGER DEFAULT 10 CHECK (low_stock_threshold >= 0),
  attributes JSONB, -- Custom product attributes (e.g., { "color": "red", "size": "M" })
  version INTEGER NOT NULL DEFAULT 1, -- Optimistic locking
  created_by UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  deleted_at TIMESTAMP -- Soft delete
);

CREATE INDEX idx_products_sku ON products(sku);
CREATE INDEX idx_products_status ON products(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_created_at ON products(created_at DESC);
CREATE INDEX idx_products_stock ON products(stock) WHERE stock < low_stock_threshold;
CREATE INDEX idx_products_search ON products USING GIN(to_tsvector('english', name || ' ' || COALESCE(description, '')));
```

**Important Columns**:
- `sku`: Unique product identifier (enforced by unique constraint)
- `status`: Draft or published (controls visibility in public catalog)
- `stock`: Current inventory level
- `low_stock_threshold`: Triggers low-stock alert when stock falls below this value
- `attributes`: JSONB column for custom product attributes (e.g., color, size, material)
- `version`: Optimistic locking to prevent concurrent edit conflicts
- `deleted_at`: Soft delete timestamp (NULL if not deleted)

#### `categories` Table

```sql
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  parent_id UUID REFERENCES categories(id) ON DELETE CASCADE,
  order INTEGER NOT NULL DEFAULT 0, -- Display order within parent
  created_by UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE(name, parent_id) -- Category name unique per parent
);

CREATE INDEX idx_categories_parent_id ON categories(parent_id);
CREATE INDEX idx_categories_order ON categories(parent_id, order);
```

**Important Columns**:
- `parent_id`: Self-referential foreign key for hierarchical structure (NULL for root categories)
- `order`: Display order within parent category (used for drag-and-drop reordering)
- `UNIQUE(name, parent_id)`: Ensures category name is unique per parent (allows same name in different parents)

#### `product_categories` Table

```sql
CREATE TABLE product_categories (
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  category_id UUID NOT NULL REFERENCES categories(id) ON DELETE CASCADE,
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  PRIMARY KEY (product_id, category_id)
);

CREATE INDEX idx_product_categories_category_id ON product_categories(category_id);
```

**Important Columns**:
- `(product_id, category_id)`: Composite primary key for many-to-many relationship
- `ON DELETE CASCADE`: Automatically delete join table rows when product or category is deleted

#### `product_images` Table

```sql
CREATE TABLE product_images (
```

**Important Columns**:

#### `users` Table

```sql
CREATE TABLE users (
```

### Query and Index Strategy

**High-Frequency Reads**:
- Product list (paginated, filtered by category, price, status, search query)
- Category tree (hierarchical, ordered)
- Audit logs (paginated, filtered by user, action, date range)
- Bulk operation job status (polled every 2 seconds during processing)

**Write-Heavy Paths**:
- Product creation, editing, deletion (CRUD operations)
- Audit log appends (every administrative action)
- Bulk operation job status updates (every batch processed)

**Suggested Indexes**:

| Table | Index | Purpose | Type |
|---|---|---|---|
| `products` | `idx_products_sku` | Enforce SKU uniqueness, fast lookup by SKU | B-tree |
| `products` | `idx_products_status` | Filter by status (draft/published), exclude soft-deleted | Partial B-tree |
| `products` | `idx_products_created_at` | Sort by creation date (newest first) | B-tree (DESC) |
| `products` | `idx_products_stock` | Find low-stock products for alerts | Partial B-tree |
| `products` | `idx_products_search` | Full-text search on name and description | GIN (tsvector) |
| `categories` | `idx_categories_parent_id` | Query child categories by parent | B-tree |
| `categories` | `idx_categories_order` | Sort categories by order within parent | B-tree |
| `product_categories` | `idx_product_categories_category_id` | Query products by category | B-tree |
| `product_images` | `idx_product_images_product_id` | Query images by product, ordered | B-tree |
| `users` | `idx_users_email` | Fast lookup by email (login) | B-tree |

**Query Optimization**:
- Use `LIMIT` and `OFFSET` for pagination (or cursor-based pagination for large datasets)
- Use `WHERE deleted_at IS NULL` to exclude soft-deleted products
- Use `to_tsvector` and `to_tsquery` for full-text search (PostgreSQL)
- Use `EXPLAIN ANALYZE` to identify slow queries and missing indexes

### Migration Strategy

**Initial Schema Rollout**:
- Use Prisma Migrate or Flyway for schema versioning
- Initial migration creates all tables, indexes, and constraints
- Seed database with default admin user (synced from Cognito)

**Future Evolution Approach**:
- All schema changes versioned via migrations (no manual SQL)
- Backward-compatible changes (add column, add index) applied immediately
- Breaking changes (rename column, drop table) require multi-step migration:
  1. Add new column/table alongside old
  2. Backfill data
  3. Update application code to use new column/table
  4. Drop old column/table in next release

**Backfill and Compatibility Concerns**:
- Custom product attributes (JSONB column) added in later migration; existing products have `NULL` attributes
- Soft-delete column (`deleted_at`) added in later migration; existing products have `NULL` (not deleted)
- Optimistic locking (`version` column) added in later migration; existing products have `version = 1`

### Reliability and Governance

**Backups and Restore Expectations**:
- **RDS**: Automated daily backups (7-day retention); point-in-time recovery within 7 days
- **S3**: Versioning enabled for product images; lifecycle policy transitions old versions to Glacier after 90 days
- **DynamoDB**: Point-in-time recovery enabled for audit logs and bulk operation jobs (35-day retention)

**Retention Requirements**:
- **Products**: Retained indefinitely (unless hard-deleted by admin)
- **Soft-deleted products**: Retained for 90 days, then hard-deleted via scheduled job
- **Audit logs**: Retained for 2 years in DynamoDB; archived to S3 after 1 year for cost optimization
- **Bulk operation jobs**: Retained for 30 days in DynamoDB; auto-deleted via TTL

**Access Controls and Sensitive Data Handling**:
- **PII**: User email and name stored in `users` table; encrypted at rest (AES-256)
- **Audit logs**: May contain PII in `changes` field; access restricted to admin role
- **Database access**: RDS accessible only from API Lambda/Fargate (no public access); IAM authentication for RDS connections
- **Secrets**: RDS password stored in AWS Secrets Manager; rotated automatically every 90 days

### Open Questions

1. **Category Hierarchy Depth Limit**: Should we enforce a maximum depth (e.g., 5 levels) via database constraint or application logic? **Recommendation**: Application logic (more flexible).
2. **Product Attributes Schema**: Should we use JSONB column or EAV (Entity-Attribute-Value) model? **Recommendation**: JSONB (simpler, better performance for read-heavy queries).
3. **Audit Log Archival**: Should we archive audit logs to S3 after 1 year or keep in DynamoDB? **Recommendation**: Archive to S3 for cost optimization; DynamoDB on-demand pricing is expensive for long-term storage.
4. **Soft-Delete Retention**: Should soft-deleted products be retained for 90 days or indefinitely? **Recommendation**: 90 days (balances recoverability with storage cost).

## Integration & Background Jobs

### Image Processing

**Trigger**: S3 upload event (via presigned URL)

**Responsibility**:
- Resize uploaded image to multiple dimensions (thumbnail: 150x150, medium: 600x600, large: 1200x1200)
- Store resized images in S3 with versioned URLs (e.g., `images/abc123-thumbnail.jpg?v=1`)
- Update `product_images` table with CloudFront URLs

**Implementation**:
- Lambda function (Node.js 20) with Sharp library
- Triggered by S3 `ObjectCreated` event
- Timeout: 5 minutes (sufficient for 50 MB images)
- Memory: 1024 MB (faster processing)

**Failure Handling**:
- Retry 3 times with exponential backoff (1s, 2s, 4s)
- Send failed events to DLQ
- Alert admin via SNS if DLQ message count > 0

**Observability**:
- CloudWatch Logs: Log image size, processing time, errors
- X-Ray tracing: Trace S3 upload → Lambda → S3 resized images
- Custom metric: `ImageProcessingDuration` (p95, p99)

**Trigger**: S3 upload event (via presigned URL)

**Responsibility**:
- Scan uploaded image with ClamAV or AWS-native solution (e.g., Trend Micro File Security)
- Delete infected files from S3
- Log scan result to CloudWatch Logs
- Alert admin via SNS if malware detected

**Implementation**:
- Lambda function (Node.js 20) with ClamAV Lambda layer or third-party scanner
- Triggered by S3 `ObjectCreated` event
- Timeout: 2 minutes
- Memory: 512 MB

**Failure Handling**:
- Retry 3 times with exponential backoff
- Send failed events to DLQ
- Alert admin via SNS if DLQ message count > 0

**Observability**:
- CloudWatch Logs: Log scan result (clean, infected, error)

### Low-Stock Alerts

**Trigger**: EventBridge scheduled rule (every 1 hour)

**Responsibility**:
- Query `products` table for products with `stock < low_stock_threshold`

**Implementation**:
- Lambda function (Node.js 20) with Prisma ORM
- Triggered by EventBridge rule (`rate(1 hour)`)

**Failure Handling**:

**Observability**:
- CloudWatch Logs: Log low-stock products, alert sent

### Bulk Operation Worker

**Trigger**: SQS message (from bulk operation API)

**Implementation**:
- Lambda function (Node.js 20) with Prisma ORM

- CloudFront enforces HTTPS

- SQS: AES-256 encryption (KMS)
- Secrets Manager: AES-256 encryption (KMS)

### Role-Based Access Control (RBAC)

**Roles**:
- **Editor**: Can create, edit products and categories; cannot delete or perform bulk operations
- **Manager**: Can create, edit, soft-delete products and categories; can perform bulk operations
- **Admin**: Full access (hard-delete, manage users, view audit logs)

**Enforcement**:
- API Gateway authorizer validates Cognito JWT tokens
- Lambda functions check user role (from Cognito groups claim) and enforce permissions server-side
- No client-side authorization (all permissions checked server-side)

**What is Logged**:
- All product CRUD operations (create, update, delete)
- All category CRUD operations (create, update, delete)
- All bulk operations (edit, delete, categorize)
- User login, logout, MFA enable/disable
- User profile updates

**Log Format**:
```json
{
  "id": "UUID",
  "user_id": "UUID",
  "user_email": "admin@example.com",
  "action": "UPDATE_PRODUCT",
  "resource_type": "product",
  "resource_id": "UUID",
  "changes": {
    "before": { "name": "Old Name", "price": 10.00 },
    "after": { "name": "New Name", "price": 15.00 }
  },
  "timestamp": "2025-01-15T12:34:56Z",
  "ip_address": "192.168.1.1",
  "user_agent": "Mozilla/5.0..."

```

**Storage**:
- DynamoDB (append-only, immutable)
- Retained for 2 years
- Archived to S3 after 1 year for cost optimization

**Access Control**:
- Only admins can view audit logs
- Audit logs cannot be modified or deleted (immutable)

**API Gateway Throttling**:
- 100 requests per minute per user (based on Cognito user ID)

## Design Decisions & Trade-offs

### Key Design Decisions

1. **Modular Monolith over Microservices**:
   - **Rationale**: Simpler deployment, shared domain logic, easier debugging for MVP
   - **Trade-off**: Harder to scale individual components independently; can extract microservices later if needed

2. **Lambda over ECS Fargate**:
   - **Rationale**: Serverless, auto-scaling, pay-per-use, lower operational complexity
   - **Trade-off**: Cold start latency (mitigated with provisioned concurrency); 15-minute timeout limit

3. **PostgreSQL over DynamoDB for Primary Database**:
   - **Rationale**: Relational model fits product-category hierarchy; ACID transactions; full-text search
   - **Trade-off**: Harder to scale horizontally; requires read replicas for read-heavy queries

4. **DynamoDB for Audit Logs**:
   - **Rationale**: Append-only, immutable, scalable, point-in-time recovery
   - **Trade-off**: More expensive for long-term storage; archived to S3 after 1 year

5. **React over Vue/Angular**:
   - **Rationale**: Largest ecosystem, best TypeScript support, most AWS Amplify integrations
   - **Trade-off**: More boilerplate than Vue; steeper learning curve than Vue

6. **Zustand over Redux Toolkit**:
   - **Rationale**: Simpler, less boilerplate, sufficient for this app's state complexity
   - **Trade-off**: Less mature ecosystem than Redux; fewer DevTools integrations

7. **TanStack Query over Manual Fetch**:
   - **Rationale**: Automatic caching, refetching, optimistic updates reduce boilerplate and improve UX
   - **Trade-off**: Learning curve; adds dependency

8. **Tailwind CSS over CSS-in-JS**:
   - **Rationale**: Faster, smaller bundle size, easier to maintain for utility-first design
   - **Trade-off**: Less dynamic styling; requires build step

9. **Presigned URLs for Image Upload**:
   - **Rationale**: Direct upload to S3 reduces API load; faster upload for large files
   - **Trade-off**: More complex client-side logic; requires S3 event trigger for processing

10. **Optimistic Locking for Concurrent Edits**:
    - **Rationale**: Prevents data loss from concurrent edits; user-friendly conflict resolution
    - **Trade-off**: Requires version field in database; adds complexity to update logic

### Trade-offs

| Decision | Benefit | Cost | Mitigation |
|---|---|---|---|
| Lambda over ECS Fargate | Lower cost, auto-scaling, simpler ops | Cold start latency | Provisioned concurrency for critical endpoints |
| PostgreSQL over DynamoDB | Relational model, ACID, full-text search | Harder to scale horizontally | Use read replicas for read-heavy queries |
| DynamoDB for Audit Logs | Scalable, immutable, point-in-time recovery | Expensive for long-term storage | Archive to S3 after 1 year |
| Presigned URLs for Image Upload | Faster upload, reduced API load | More complex client-side logic | Provide clear error messages, retry logic |
| Optimistic Locking | Prevents data loss from concurrent edits | Adds complexity to update logic | Show user-friendly conflict resolution UI |
| Client-side Validation Only | Faster UX | Security risk if server-side validation missing | Always enforce server-side validation |
| Lazy Loading for Routes | Reduces initial bundle size | Slight delay when navigating to new pages | Prefetch critical routes on hover |
| Soft Delete | Recoverability | Storage cost for deleted products | Hard-delete after 90 days via scheduled job |
