## Application Architecture

### Architecture Pattern

**Modular Monolith with Service-Layer Separation**

The system follows a **modular monolith** architecture for the MVP, with clear boundaries between frontend, backend API, and background jobs. This approach balances simplicity (single deployment unit) with modularity (independent services can be extracted to microservices later if needed).

**Key Architectural Decisions:**

1. **Frontend**: Single-page React application served via CloudFront, communicates with backend via REST API
2. **Backend**: NestJS API deployed on Lambda (serverless) or ECS Fargate (containerized), with modular services (ProductService, CategoryService, AuditLogService, etc.)
3. **Database**: RDS PostgreSQL for relational data (products, categories, users), DynamoDB for audit logs and bulk job status
4. **Background Jobs**: Lambda functions triggered by SQS messages (bulk operations), S3 events (image processing), or EventBridge schedules (alerts, exports)
5. **API Gateway**: Single REST API with versioned endpoints (`/api/v1/`), Cognito authorizer for authentication, request validation, and throttling

**Future Extensibility:**

- Custom product attributes stored in JSON column (PostgreSQL JSONB)
- Advanced search via Amazon OpenSearch (if PostgreSQL full-text search is insufficient)
- Microservices extraction (e.g., ImageService, BulkOperationService) if scale requires independent deployment

### Page Designs

#### LoginPage
- **Route**: `/login`
- **Purpose**: Admin authentication with email/password and optional MFA
- **Components**: LoginForm (email, password, MFA code), ErrorAlert, LoadingSpinner
- **State**: Form state (email, password, mfaCode), loading state, error state
- **API Calls**: `POST /auth/login` (Cognito)
- **User Interactions**: Enter credentials → submit → MFA challenge (if enabled) → redirect to dashboard on success

#### DashboardPage
- **Route**: `/`
- **Purpose**: Overview of product catalog with search, filter, and quick actions
- **Components**: ProductListTable, SearchBar, FilterPanel, QuickStats, BulkActionsToolbar
- **State**: Product list (server cache via TanStack Query), selected products (local state), filters (URL query params)
- **API Calls**: `GET /api/v1/products?page=1&limit=50&search=...&category_id=...`
- **User Interactions**: Search products → apply filters → select products → bulk actions → navigate to create/edit

#### ProductCreatePage / ProductEditPage
- **Route**: `/products/new`, `/products/:id/edit`
- **Purpose**: Create or edit product with form validation and image upload
- **Components**: ProductForm, ImageUploader, CategoryMultiSelect, FormActions
- **State**: Form state (React Hook Form), image upload state (presigned URLs, upload progress), validation errors
- **API Calls**: 
  - `POST /api/v1/images/presigned-url` (get presigned URL)
  - Direct S3 upload via presigned URL
  - `POST /api/v1/products` (create) or `PUT /api/v1/products/:id` (update)
- **User Interactions**: Fill form → upload images → select categories → submit → success message → redirect to product list

#### CategoriesPage
- **Route**: `/categories`
- **Purpose**: Manage category hierarchy with drag-and-drop reordering
- **Components**: CategoryTree (react-dnd or react-beautiful-dnd), CategoryForm (modal), DeleteConfirmDialog
- **State**: Category tree (server cache), selected category (local state), drag state
- **API Calls**: `GET /api/v1/categories`, `POST /api/v1/categories`, `PUT /api/v1/categories/:id`, `DELETE /api/v1/categories/:id`
- **User Interactions**: View tree → create category → edit category → drag to reorder → delete category (with confirmation)

#### AuditLogsPage
- **Route**: `/audit-logs`
- **Purpose**: View audit trail of all administrative actions
- **Components**: AuditLogTable, FilterPanel (user, action, date range), ExportButton
- **State**: Audit logs (server cache), filters (URL query params)
- **API Calls**: `GET /api/v1/audit-logs?page=1&limit=50&user_id=...&action=...`, `POST /api/v1/audit-logs/export`
- **User Interactions**: View logs → apply filters → export to CSV

#### BulkOperationsModal
- **Route**: Modal overlay on DashboardPage
- **Purpose**: Perform bulk actions on selected products
- **Components**: ProductSelectionList, ActionSelector, ProgressIndicator, ConfirmationDialog
- **State**: Selected products (from parent), bulk action (edit/delete/categorize), job status (polling)
- **API Calls**: `POST /api/v1/products/bulk`, `GET /api/v1/products/bulk/:job_id/status` (polling)
- **User Interactions**: Select products → choose action → confirm → monitor progress → completion notification

#### Global State (Zustand)

| selectedProducts | `Set<productId>` | Checkbox selection on dashboard |

#### Server State (TanStack Query)

**Queries** (data fetched from API):
- `products` — paginated product list with filters
- `product/:id` — single product details
- `categories` — hierarchical category tree
- `auditLogs` — paginated audit logs with filters
- `user/me` — current user profile

**Mutations** (data modified via API):
- `createProduct` — create new product
- `updateProduct` — update existing product
- `deleteProduct` — soft-delete or hard-delete product
- `createCategory` — create new category
- `updateCategory` — update category
- `deleteCategory` — delete category
- `bulkOperation` — enqueue bulk operation job

**Cache Strategy**:
- **Stale-while-revalidate**: Product list, categories (refetch on window focus)
- **Optimistic updates**: Product edit, category reorder (update cache immediately, rollback on error)
- **Cache invalidation**: After create/update/delete, invalidate related queries (e.g., invalidate `products` after `createProduct`)

### API Integration Layer

#### API Service Structure

```typescript
// src/services/api.ts
import axios from 'axios';
import { getAuthToken, refreshAuthToken } from './auth';

const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL, // e.g., https://api.example.com/api/v1
  timeout: 30000,
});

// Request interceptor: inject auth token
apiClient.interceptors.request.use((config) => {
  const token = getAuthToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// Response interceptor: handle 401 (token expired), retry with refresh
apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      const newToken = await refreshAuthToken();
      if (newToken) {
        error.config.headers.Authorization = `Bearer ${newToken}`;
        return apiClient.request(error.config); // Retry request
      }
    }
    return Promise.reject(error);
  }
);

export default apiClient;
```

### Design Decisions & Trade-offs

1. **React over Vue/Angular**: React chosen for larger ecosystem, TypeScript support, and team familiarity. Trade-off: More boilerplate than Vue, but better long-term maintainability.

2. **Zustand over Redux Toolkit**: Zustand chosen for simplicity and smaller bundle size. Trade-off: Less opinionated than Redux, but sufficient for this use case.

3. **TanStack Query over manual fetch**: TanStack Query chosen for automatic caching, refetching, and optimistic updates. Trade-off: Learning curve, but significantly reduces boilerplate and improves UX.

4. **Tailwind CSS over CSS Modules**: Tailwind chosen for rapid prototyping and consistent design system. Trade-off: Larger HTML files, but faster development and easier maintenance.

5. **Vite over Create React App**: Vite chosen for faster dev server and optimized production builds. Trade-off: Newer tool, but better performance and developer experience.

6. **Client-side routing over server-side**: React Router chosen for SPA experience. Trade-off: Initial load time, but better perceived performance after first load.

## Backend API Design

### Service Scope

**Responsibilities:**
- Authenticate and authorize admin users via AWS Cognito
- Provide REST API for product, category, and audit log CRUD operations
- Validate all incoming requests (schema, business rules)
- Enforce role-based access control (RBAC) server-side
- Process image uploads (presigned URLs, malware scanning, resizing)
- Enqueue and track bulk operation jobs
- Log all administrative actions to audit trail
- Trigger background jobs (low-stock alerts, scheduled exports)

**Non-Goals:**
- Public-facing storefront (separate application consumes catalog via API)
- Real-time collaboration (e.g., live cursors, chat)
- Advanced analytics or BI dashboards
- Payment processing or order management

**External Dependencies:**
- AWS Cognito (authentication, user management)
- Amazon RDS PostgreSQL (product, category, user data)
- Amazon S3 (image storage)
- Amazon CloudFront (image CDN)
- Amazon SQS (bulk operation job queue)
- Amazon DynamoDB (audit logs, bulk job status)
- Amazon EventBridge (scheduled jobs)
- Public storefront API (consumes product catalog data)

### Domain Model

**Core Entities:**

1. **Product**
   - Attributes: id, name, description, price, sku, status (draft/published), stock, created_at, updated_at, deleted_at, version
   - Relationships: Belongs to multiple categories (many-to-many), has many images (one-to-many), owned by creator (user_id)
   - Invariants: SKU must be unique, price must be positive, stock must be non-negative, version increments on each update

2. **Category**
   - Attributes: id, name, parent_id, order, created_at, updated_at
   - Relationships: Self-referential (parent-child hierarchy), has many products (many-to-many)
   - Invariants: Name must be unique per parent, hierarchy depth must not exceed 5 levels, order must be unique per parent

3. **ProductCategory** (join table)
   - Attributes: product_id, category_id
   - Relationships: Links products to categories
   - Invariants: Composite primary key (product_id, category_id)

4. **ProductImage**
   - Attributes: id, product_id, image_url, order, created_at
   - Relationships: Belongs to product
   - Invariants: Order must be unique per product, image_url must be valid S3 URL

5. **AuditLog**
   - Attributes: id, user_id, action, resource_type, resource_id, changes (JSON), timestamp
   - Relationships: Belongs to user, references product/category/other resource
   - Invariants: Immutable (append-only), timestamp must be in UTC

6. **User**
   - Attributes: id, email, name, role (editor/manager/admin), mfa_enabled, created_at, updated_at
   - Relationships: Owns products, categories, audit logs
   - Invariants: Email must be unique, role must be one of (editor, manager, admin)

7. **BulkOperationJob**
   - Attributes: job_id, user_id, action, product_ids, params, status, progress, created_at, updated_at
   - Relationships: Belongs to user, references products
   - Invariants: Status must be one of (pending, processing, completed, failed), progress must be 0-100

**Validation Rules:**

- **Product**: Name required (max 255 chars), description optional (max 5000 chars), price > 0, SKU required (unique, alphanumeric, max 50 chars), stock >= 0, status in (draft, published)
- **Category**: Name required (max 255 chars, unique per parent), parent_id optional (must reference existing category), order >= 0
- **ProductImage**: Image file type in (JPG, PNG, WebP), file size <= 50MB, max 20 images per product
- **AuditLog**: All fields required, changes must be valid JSON
- **User**: Email must be valid format, role in (editor, manager, admin)

#### Service Layer

**Business logic, validation, and orchestration.**

```typescript
// src/products/products.service.ts
@Injectable()
export class ProductsService {
  constructor(
    private readonly productsRepository: ProductsRepository,
    private readonly auditLogService: AuditLogService,
  ) {}

  async createProduct(dto: CreateProductDto, user: User): Promise<Product> {
    // Validate SKU uniqueness
    const existingProduct = await this.productsRepository.findBySku(dto.sku);
    if (existingProduct) {
      throw new ConflictException('SKU already exists');
    }

    // Create product
    const product = await this.productsRepository.create({
      ...dto,
      created_by: user.id,
      version: 1,
    });

    // Log audit event
    await this.auditLogService.log({
      user_id: user.id,
      action: 'CREATE_PRODUCT',
      resource_type: 'product',
      resource_id: product.id,
      changes: dto,
    });

    return product;
  }

  async updateProduct(id: string, dto: UpdateProductDto, user: User): Promise<Product> {
    // Fetch existing product
    const product = await this.productsRepository.findById(id);
    if (!product) {
      throw new NotFoundException('Product not found');
    }

    // Optimistic locking: check version
    if (dto.version !== product.version) {
      throw new ConflictException('Product has been modified by another user');
    }

    // Update product
    const updatedProduct = await this.productsRepository.update(id, {
      ...dto,
      version: product.version + 1,
    });

    // Log audit event
    await this.auditLogService.log({
      user_id: user.id,
      action: 'UPDATE_PRODUCT',
      resource_type: 'product',
      resource_id: id,
      changes: dto,
    });

    return updatedProduct;
  }
}
```

```typescript
// src/products/products.repository.ts
@Injectable()
export class ProductsRepository {
  constructor(private readonly db: DatabaseService) {}

  async findById(id: string): Promise<Product | null> {
    return this.db.query('SELECT * FROM products WHERE id = $1 AND deleted_at IS NULL', [id]);
  }

  async findBySku(sku: string): Promise<Product | null> {
    return this.db.query('SELECT * FROM products WHERE sku = $1 AND deleted_at IS NULL', [sku]);
  }

  async create(data: CreateProductData): Promise<Product> {
    return this.db.query(
      'INSERT INTO products (name, description, price, sku, status, stock, created_by, version) VALUES ($1, $2, $3, $4, $5, $6, $7, $8) RETURNING *',
      [data.name, data.description, data.price, data.sku, data.status, data.stock, data.created_by, data.version]
    );
  }

  async update(id: string, data: UpdateProductData): Promise<Product> {
    return this.db.query(
      'UPDATE products SET name = $1, description = $2, price = $3, sku = $4, status = $5, stock = $6, version = $7, updated_at = NOW() WHERE id = $8 RETURNING *',
      [data.name, data.description, data.price, data.sku, data.status, data.stock, data.version, id]
    );
  }

  async softDelete(id: string): Promise<void> {
    await this.db.query('UPDATE products SET deleted_at = NOW() WHERE id = $1', [id]);
  }

  async hardDelete(id: string): Promise<void> {
    await this.db.query('DELETE FROM products WHERE id = $1', [id]);
  }
}
```

#### Authentication and Authorization

**Authentication (AWS Cognito):**
- Admin users authenticate via Cognito user pool
- Email/password login with optional MFA (TOTP or SMS)
- Cognito returns JWT access token (30-min expiry) and refresh token (7-day expiry)
- Frontend stores tokens in memory (not localStorage for security)
- API Gateway authorizer validates JWT signature and expiry

**Authorization (Role-Based Access Control):**
- Cognito groups: `editors`, `managers`, `admins`
- API Gateway authorizer extracts user role from JWT claims
- NestJS `RoleGuard` enforces permissions server-side
- Permission matrix:
  - **Editor**: Create, edit products and categories; view audit logs (own actions only)
  - **Manager**: All editor permissions + delete products (soft-delete), bulk operations
  - **Admin**: All manager permissions + hard-delete products, manage users, view all audit logs, export audit logs

**Session Management:**
- Access tokens expire after 30 minutes
- Frontend refreshes tokens automatically using refresh token
- Refresh tokens rotate every 15 minutes (Cognito advanced security)
- Sessions expire after 30 minutes of inactivity (Cognito session timeout)

3. **Optimistic Locking vs. Pessimistic Locking**: Should concurrent edits use optimistic locking (version field) or pessimistic locking (row-level locks)? **Recommendation**: Optimistic locking (better performance, acceptable UX with conflict resolution).

**Core Domains:**
- **Product Management**: Products, categories, product-category relationships, product images

- Categories owned by creator (user_id)

### Entities and Relationships

| Entity | Purpose | Primary Key | Relationships |
|---|---|---|---|
| products | Store product data | id (UUID) | Belongs to multiple categories (many-to-many via product_categories), has many images (one-to-many via product_images), owned by user (user_id) |
| categories | Store category hierarchy | id (UUID) | Self-referential (parent_id), has many products (many-to-many via product_categories) |
| product_categories | Join table for products and categories | (product_id, category_id) | Links products to categories |

| bulk_operation_jobs | Track bulk operation status | job_id (UUID) | Belongs to user (user_id), references products (product_ids array) |

### Schema Notes

#### products Table

```sql
CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  description TEXT,
  price DECIMAL(10, 2) NOT NULL CHECK (price > 0),
  sku VARCHAR(50) NOT NULL UNIQUE,
  status VARCHAR(20) NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published')),
  stock INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0),
  attributes JSONB, -- Custom product attributes (e.g., color, size, material)
  created_by UUID NOT NULL REFERENCES users(id),
  version INTEGER NOT NULL DEFAULT 1, -- Optimistic locking
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  deleted_at TIMESTAMP -- Soft delete
);

CREATE INDEX idx_products_status ON products(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_sku ON products(sku) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_created_at ON products(created_at DESC) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_price ON products(price) WHERE deleted_at IS NULL;
CREATE INDEX idx_products_stock ON products(stock) WHERE deleted_at IS NULL AND stock < 10; -- Low-stock index
CREATE INDEX idx_products_attributes ON products USING GIN(attributes); -- JSONB index for custom attributes
```

**Important Columns:**
- `sku`: Unique product identifier (alphanumeric, max 50 chars)
- `status`: Draft (hidden from storefront) or published (visible)
- `stock`: Current inventory level (triggers low-stock alerts)
- `attributes`: JSONB column for custom product attributes (e.g., `{"color": "red", "size": "L"}`)
- `version`: Optimistic locking field (increments on each update)
- `deleted_at`: Soft delete timestamp (NULL if active)

**Constraints:**
- `price > 0`: Price must be positive
- `stock >= 0`: Stock cannot be negative
- `status IN ('draft', 'published')`: Only two valid statuses
- `sku UNIQUE`: SKU must be unique across all products

#### categories Table

```sql
CREATE TABLE categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  parent_id UUID REFERENCES categories(id) ON DELETE CASCADE,
  order INTEGER NOT NULL DEFAULT 0, -- Display order within parent
  created_by UUID NOT NULL REFERENCES users(id),
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
  UNIQUE(parent_id, name), -- Name must be unique per parent
  UNIQUE(parent_id, order) -- Order must be unique per parent
);

CREATE INDEX idx_categories_parent_id ON categories(parent_id);
CREATE INDEX idx_categories_order ON categories(parent_id, order);
```

**Important Columns:**
- `parent_id`: Self-referential foreign key for hierarchy (NULL for root categories)
- `order`: Display order within parent (used for drag-and-drop reordering)

**Constraints:**
- `UNIQUE(parent_id, name)`: Category name must be unique per parent
- `UNIQUE(parent_id, order)`: Order must be unique per parent
- `ON DELETE CASCADE`: Deleting a category cascades to child categories and product-category assignments

**DynamoDB Table Schema:**

- **Partition Key**: `id` (UUID)
- **Sort Key**: `timestamp` (ISO 8601 timestamp)
- **Attributes**:

**Global Secondary Index (GSI):**
- **GSI Name**: `user_id-timestamp-index`
- **Partition Key**: `user_id`
- **Sort Key**: `timestamp`
- **Purpose**: Query audit logs by user

**Why DynamoDB?**
- Immutable, append-only logs (no updates or deletes)
- High write throughput for audit events
- Scalable to billions of records
- Point-in-time recovery for disaster recovery
- Cost-effective for infrequent reads (on-demand billing)

**DynamoDB Table Schema:**

- **Partition Key**: `job_id` (UUID)
- **Attributes**:
  - `user_id` (UUID)

  - `product_ids` (List of UUIDs)
  - `params` (Map, JSON object with action-specific parameters)

**Global Secondary Index (GSI):**
- **GSI Name**: `user_id-created_at-index`
- **Partition Key**: `user_id`
- **Sort Key**: `created_at`
- **Purpose**: Query jobs by user

**Why DynamoDB?**
- Fast reads/writes for job status updates
- Scalable to handle high concurrency
- TTL for automatic cleanup of old jobs (30-day retention)

### Query and Index Strategy

#### High-Frequency Reads

1. **Product List (Dashboard)**
   - Query: `SELECT * FROM products WHERE deleted_at IS NULL ORDER BY created_at DESC LIMIT 50 OFFSET 0`
   - Index: `idx_products_created_at` (B-tree index on `created_at DESC`)
   - Optimization: Cache results in ElastiCache (Redis) with 5-minute TTL

2. **Product Search**
   - Query: `SELECT * FROM products WHERE deleted_at IS NULL AND to_tsvector('english', name || ' ' || description) @@ to_tsquery('search_term')`
   - Index: `CREATE INDEX idx_products_fts ON products USING GIN(to_tsvector('english', name || ' ' || description)) WHERE deleted_at IS NULL`
   - Optimization: Use PostgreSQL full-text search with GIN index; consider OpenSearch for advanced search

3. **Category Tree**
   - Query: `SELECT * FROM categories ORDER BY parent_id, order`
   - Index: `idx_categories_parent_id`, `idx_categories_order`
   - Optimization: Cache entire tree in ElastiCache (Redis) with 10-minute TTL

4. **Product Details**
   - Query: `SELECT * FROM products WHERE id = $1 AND deleted_at IS NULL`
   - Index: Primary key index on `id`
   - Optimization: Cache individual products in ElastiCache (Redis) with 5-minute TTL

#### Write-Heavy Paths

1. **Product Creation**
   - Insert: `INSERT INTO products (...) VALUES (...)`
   - Index Impact: Updates `idx_products_status`, `idx_products_sku`, `idx_products_created_at`
   - Optimization: Batch inserts for bulk operations (100 products per batch)

2. **Product Update**
   - Update: `UPDATE products SET ... WHERE id = $1 AND version = $2`
   - Index Impact: Updates `idx_products_status`, `idx_products_price`, `idx_products_stock`
   - Optimization: Optimistic locking with version field to prevent concurrent edit conflicts

3. **Audit Log Insert**
   - Insert: DynamoDB `PutItem` operation
   - Index Impact: None (DynamoDB auto-indexes partition key and sort key)
   - Optimization: Batch writes for bulk operations (25 items per batch)

#### Suggested Indexes and Why

| Index | Purpose | Type | Notes |
|---|---|---|---|
| `idx_products_status` | Filter by status (draft/published) | B-tree | Partial index (WHERE deleted_at IS NULL) |
| `idx_products_sku` | Enforce SKU uniqueness, search by SKU | B-tree | Partial index (WHERE deleted_at IS NULL) |
| `idx_products_created_at` | Sort by creation date (newest first) | B-tree | Partial index (WHERE deleted_at IS NULL) |
| `idx_products_price` | Filter by price range | B-tree | Partial index (WHERE deleted_at IS NULL) |
| `idx_products_stock` | Low-stock alerts | B-tree | Partial index (WHERE deleted_at IS NULL AND stock < 10) |
| `idx_products_attributes` | Search custom attributes | GIN | JSONB index for flexible queries |
| `idx_products_fts` | Full-text search on name and description | GIN | PostgreSQL full-text search |
| `idx_categories_parent_id` | Query child categories | B-tree | Supports hierarchical queries |
| `idx_categories_order` | Sort categories by order | B-tree | Composite index (parent_id, order) |
| `idx_product_categories_category_id` | Query products by category | B-tree | Supports many-to-many joins |
| `idx_product_images_product_id` | Query images by product | B-tree | Supports one-to-many joins |

### Image Processing

**Trigger**: S3 upload event (via presigned URL)

**Responsibility**:
- Lambda function resizes images to multiple dimensions (thumbnail: 150x150, medium: 600x600, large: 1200x1200)
- Stores resized images in S3 (same bucket, different prefix)
- Updates `product_images` table with metadata (image_url, order)

**Failure Handling**:
- Retry 3 times with exponential backoff (1s, 2s, 4s)
- Send failed events to Dead Letter Queue (DLQ)
- Alert admin via SNS on DLQ message

**Observability**:
- CloudWatch Logs: Log processing time, image dimensions, errors
- X-Ray: Trace Lambda invocation, S3 operations, database writes
- Custom Metric: `ImageProcessingDuration` (milliseconds)

**Implementation**:
```typescript
// Lambda function: image-processing
import { S3Event } from 'aws-lambda';
import sharp from 'sharp';
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { DatabaseService } from './database.service';

export async function handler(event: S3Event) {
  const s3 = new S3Client({});
  const db = new DatabaseService();

  for (const record of event.Records) {
    const bucket = record.s3.bucket.name;
    const key = record.s3.object.key;

    // Download original image from S3
    const originalImage = await s3.getObject({ Bucket: bucket, Key: key });

    // Resize to multiple dimensions
    const sizes = [
      { name: 'thumbnail', width: 150, height: 150 },
      { name: 'medium', width: 600, height: 600 },
      { name: 'large', width: 1200, height: 1200 },
    ];

    for (const size of sizes) {
      const resizedImage = await sharp(originalImage.Body)
        .resize(size.width, size.height, { fit: 'cover' })
        .toBuffer();

      // Upload resized image to S3
      const resizedKey = `images/${size.name}/${key}`;
      await s3.send(new PutObjectCommand({
        Bucket: bucket,
        Key: resizedKey,
        Body: resizedImage,
        ContentType: 'image/jpeg',
      }));
    }

    // Update product_images table
    const imageUrl = `https://cdn.example.com/images/large/${key}`;
    await db.query('UPDATE product_images SET image_url = $1 WHERE id = $2', [imageUrl, record.s3.object.key]);
  }
}
```

```typescript
export async function handler() {
  const db = new DatabaseService();
  const sns = new SNSClient({});

  // Query low-stock products
  const lowStockProducts = await db.query(
    'SELECT id, name, sku, stock FROM products WHERE stock < 10 AND deleted_at IS NULL'
  );

  if (lowStockProducts.length > 0) {
    // Send SNS notification
    const message = `Low-stock alert: ${lowStockProducts.length} products below threshold\n\n` +
      lowStockProducts.map(p => `${p.name} (SKU: ${p.sku}): ${p.stock} units`).join('\n');

    await sns.send(new PublishCommand({
      TopicArn: process.env.SNS_TOPIC_ARN,
      Subject: 'Low-Stock Alert',
      Message: message,
    }));

    console.log(`Low-stock alert sent for ${lowStockProducts.length} products`);
  }
}
```

### Bulk Operation Worker

**Trigger**: SQS message (from bulk operation API)

**Responsibility**:
- Lambda function processes bulk edit/delete/categorize
- Updates products in batches (100 products per batch)
- Updates job status in DynamoDB (progress, status)
- Sends completion notification via SNS

**Failure Handling**:
- Retry 3 times with exponential backoff
- Send failed messages to DLQ
- Mark job as failed in DynamoDB
- Allow manual retry via API

**Observability**:
- CloudWatch Logs: Log batch processing, errors
- X-Ray: Trace Lambda invocation, database writes
- Custom Metric: `BulkOperationDuration` (milliseconds)

**Implementation**:
```typescript
// Lambda function: bulk-operation-worker
import { SQSEvent } from 'aws-lambda';
import { DatabaseService } from './database.service';
import { DynamoDBClient, UpdateItemCommand } from '@aws-sdk/client-dynamodb';

export async function handler(event: SQSEvent) {
  const db = new DatabaseService();
  const dynamodb = new DynamoDBClient({});

  for (const record of event.Records) {
    const { job_id, action, product_ids, params } = JSON.parse(record.body);

    // Update job status to processing
    await dynamodb.send(new UpdateItemCommand({
      TableName: 'bulk_operation_jobs',
      Key: { job_id: { S: job_id } },
      UpdateExpression: 'SET #status = :status, progress = :progress',
      ExpressionAttributeNames: { '#status': 'status' },
      ExpressionAttributeValues: { ':status': { S: 'processing' }, ':progress': { N: '0' } },
    }));

    // Process products in batches
    const batchSize = 100;
    for (let i = 0; i < product_ids.length; i += batchSize) {
      const batch = product_ids.slice(i, i + batchSize);

      if (action === 'bulk_edit') {
        await db.query('UPDATE products SET price = $1 WHERE id = ANY($2)', [params.price, batch]);
      } else if (action === 'bulk_delete') {
        await db.query('UPDATE products SET deleted_at = NOW() WHERE id = ANY($1)', [batch]);
      } else if (action === 'bulk_categorize') {
        await db.query('INSERT INTO product_categories (product_id, category_id) VALUES ...', [batch, params.category_id]);
      }

      // Update progress
      const progress = Math.round(((i + batchSize) / product_ids.length) * 100);
      await dynamodb.send(new UpdateItemCommand({
        TableName: 'bulk_operation_jobs',
        Key: { job_id: { S: job_id } },
        UpdateExpression: 'SET progress = :progress',
        ExpressionAttributeValues: { ':progress': { N: progress.toString() } },
      }));
    }

    // Update job status to completed
    await dynamodb.send(new UpdateItemCommand({
      TableName: 'bulk_operation_jobs',
      Key: { job_id: { S: job_id } },
      UpdateExpression: 'SET #status = :status, progress = :progress',
      ExpressionAttributeNames: { '#status': 'status' },
      ExpressionAttributeValues: { ':status': { S: 'completed' }, ':progress': { N: '100' } },
    }));
  }
}
```

### Catalog Export

**Trigger**: API request or EventBridge scheduled rule

**Responsibility**:
- Lambda function queries products
- Generates CSV or JSON export
- Uploads to S3
- Sends SNS email with download link

**Failure Handling**:
- Retry 3 times with exponential backoff
- Log failure to CloudWatch Logs
- Manual retry via API

**Observability**:
- CloudWatch Logs: Log export size, duration, errors
- Custom Metric: `CatalogExportDuration` (milliseconds)

**Implementation**:
```typescript
// Lambda function: catalog-export
import { DatabaseService } from './database.service';
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { SNSClient, PublishCommand } from '@aws-sdk/client-sns';

export async function handler(event: { format: 'csv' | 'json' }) {
  const db = new DatabaseService();
  const s3 = new S3Client({});
  const sns = new SNSClient({});

  // Query all products
  const products = await db.query('SELECT * FROM products WHERE deleted_at IS NULL');

  // Generate export
  let exportData: string;
  if (event.format === 'csv') {
    exportData = generateCSV(products);
  } else {
    exportData = JSON.stringify(products, null, 2);
  }

  // Upload to S3
  const key = `exports/catalog-${Date.now()}.${event.format}`;
  await s3.send(new PutObjectCommand({
    Bucket: process.env.EXPORT_BUCKET,
    Key: key,
    Body: exportData,
    ContentType: event.format === 'csv' ? 'text/csv' : 'application/json',
  }));

  // Send SNS email
  const downloadUrl = `https://${process.env.EXPORT_BUCKET}.s3.amazonaws.com/${key}`;
  await sns.send(new PublishCommand({
    TopicArn: process.env.SNS_TOPIC_ARN,
    Subject: 'Catalog Export Ready',
    Message: `Your catalog export is ready: ${downloadUrl}`,
  }));

  return { download_url: downloadUrl };
}

function generateCSV(products: any[]): string {
  const header = 'id,name,description,price,sku,status,stock\n';
  const rows = products.map(p => `${p.id},${p.name},${p.description},${p.price},${p.sku},${p.status},${p.stock}`).join('\n');
  return header + rows;
}
```

## Security & Compliance

### Authentication

- **AWS Cognito User Pool**: Manages admin users, email/password authentication, MFA (TOTP or SMS)
- **Password Policy**: Minimum 8 characters, uppercase, lowercase, number, special character
- **MFA Enforcement**: Required for manager and admin roles (configurable)
- **Session Management**: Access tokens expire after 30 minutes, refresh tokens expire after 7 days, tokens rotate every 15 minutes

### Authorization

- **Role-Based Access Control (RBAC)**: Three roles (editor, manager, admin) with varying permissions
- **Cognito Groups**: `editors`, `managers`, `admins` (mapped to Cognito user pool groups)
- **API Gateway Authorizer**: Validates JWT tokens, extracts user role from claims
- **Server-Side Enforcement**: NestJS `RoleGuard` enforces permissions on all API endpoints

**Permission Matrix:**

| Action | Editor | Manager | Admin |
|---|---|---|---|
| Create product | ✅ | ✅ | ✅ |
| Edit product | ✅ | ✅ | ✅ |
| Delete product (soft) | ❌ | ✅ | ✅ |
| Delete product (hard) | ❌ | ❌ | ✅ |
| Create category | ✅ | ✅ | ✅ |
| Edit category | ✅ | ✅ | ✅ |
| Delete category | ❌ | ✅ | ✅ |
| Bulk operations | ❌ | ✅ | ✅ |
| View audit logs (own) | ✅ | ✅ | ✅ |
| View audit logs (all) | ❌ | ❌ | ✅ |
| Export audit logs | ❌ | ❌ | ✅ |
| Manage users | ❌ | ❌ | ✅ |

### Encryption

- **In Transit**: TLS 1.2+ for all API requests (API Gateway, CloudFront)
- **At Rest**: AES-256 encryption for RDS, S3, DynamoDB (AWS-managed keys via KMS)
- **Secrets Management**: AWS Secrets Manager for database credentials, API keys

### Audit Logging

- **Immutable Logs**: All administrative actions logged to DynamoDB (append-only, no updates or deletes)
- **Log Fields**: user_id, action, resource_type, resource_id, changes (JSON), timestamp
- **Retention**: 2 years in DynamoDB, archived to S3 after 1 year
- **Tamper-Evident**: DynamoDB point-in-time recovery enabled, CloudWatch Logs for system logs

### WCAG 2.1 AA Compliance

- **Semantic HTML**: Use `<button>`, `<input>`, `<label>`, `<nav>`, `<main>`, etc.
- **ARIA Labels**: Add `aria-label`, `aria-describedby`, `aria-live` for dynamic content
- **Keyboard Navigation**: All interactive elements accessible via keyboard (Tab, Enter, Escape)
- **Color Contrast**: Minimum 4.5:1 for normal text, 3:1 for large text (Tailwind CSS default palette meets WCAG AA)
- **Screen Reader Support**: Test with NVDA, JAWS, VoiceOver
- **Focus Management**: Visible focus indicators, logical tab order, focus trapping in modals

## Open Questions

1. **Data Residency Region**: Which AWS region should be used for data residency compliance (e.g., us-east-1, eu-west-1, ap-southeast-2)? **Decision Needed**: Confirm with Product Owner / Legal / Compliance.

2. **Lambda vs. ECS Fargate**: Should backend API run on Lambda (serverless, auto-scaling) or ECS Fargate (containerized, more control)? **Recommendation**: Lambda for MVP (simpler, lower cost), migrate to Fargate if cold start latency becomes an issue.

3. **PostgreSQL Full-Text Search vs. OpenSearch**: Is PostgreSQL full-text search sufficient for 1M+ products, or should we use OpenSearch from the start? **Recommendation**: Start with PostgreSQL, migrate to OpenSearch if search performance degrades.

4. **MFA Enforcement Policy**: Should MFA be required for all admin roles, or only for manager and admin roles? **Recommendation**: Required for manager and admin roles (higher privileges).

5. **Backup Retention Period**: Should RDS backups be retained for 7 days (default) or longer (e.g., 30 days)? **Recommendation**: 7 days for MVP, extend to 30 days if compliance requires.

6. **Audit Log Archival**: Should audit logs be archived to S3 after 1 year, or retained in DynamoDB for the full 2 years? **Recommendation**: Archive to S3 after 1 year for cost optimization.

7. **Acceptable Page Load Time**: Is < 2 seconds acceptable for initial load of 100k product list, or should we optimize further (e.g., virtual scrolling, server-side pagination)? **Recommendation**: < 2 seconds is acceptable; use server-side pagination and lazy loading.

8. **Budget for Accessibility Audit**: Is there budget for a third-party accessibility audit, or should we rely on automated testing only? **Recommendation**: Automated testing (axe, Lighthouse) for MVP, third-party audit for post-launch validation.

9. **Brand Guidelines and Design System**: Does a brand guideline or design system exist, or should we create one from scratch? **Recommendation**: Use Tailwind CSS default palette for MVP, create brand guidelines in Phase 2.

10. **Minimum Supported Screen Size**: What is the minimum supported screen size for tablet devices (e.g., 768px, 1024px)? **Recommendation**: 768px (iPad portrait mode).
